# 1. CV is a web document, built as a second Vite entry

Date: 2026-07-13

## Status

Accepted

## Context

The downloadable CV was a hand-made static PDF (`public/cv-mathieu-schrurs.pdf`),
visually unrelated to the site. We want the CV to match the site's design and to
stay in sync with content that already lives in the repo (`config.ts` jobs and
projects, `skills.ts`).

The site is a single-page app with no router and deploys as a static build to
Vercel. Three ways to add a styled CV route were considered:

- **react-router-dom**: a runtime dependency plus a Vercel SPA rewrite so a
  direct hit to `/cv` serves `index.html`.
- **Manual `pathname` switch** in `App.tsx`: no dependency but still needs a
  rewrite and is unconventional.
- **A second Vite entry (`cv.html`)**: a real static file at `/cv`, no router
  dependency, no rewrite config, and it can import the site's real theme tokens,
  fonts, and mixins so styling has a single source of truth.

Rebuilding the PDF in an external design tool (Canva/Figma) was also considered
but rejected: the content would drift from the repo and could not reuse the
design tokens.

## Decision

Build the CV as a React document rendered by a **second Vite entry**
(`cv.html` -> `<Resume/>`), served at `/cv`. It reuses `GlobalStyle`, the theme,
`variables.ts`, fonts, and `mixins.ts`, and carries an `@media print`
stylesheet. Content is driven by `config.ts` and `skills.ts`, extended with
CV-only structured fields (education, languages, a short summary).

The downloadable PDF is kept for recruiters, but is **regenerated by
print-to-PDF from the `/cv` page** rather than authored separately. The hero
"CV" button continues to download that PDF.

## Consequences

- The site design and the CV cannot drift apart: both consume the same tokens.
- A future reader will see two Vite entries; this ADR explains why.
- The entry lives at `cv/index.html`, so it is served at `/cv` on Vercel
  (directory index resolution, no rewrite needed). Note: the local `vite
  preview` server applies an SPA fallback and only serves it at `/cv/` (with
  the trailing slash) or `/cv/index.html`; `/cv` without the slash falls back
  to the main app locally. This is a preview-server quirk, not production.
- Regenerating the PDF is a **manual step**: after changing CV content, open
  `/cv`, print to PDF, and replace `public/cv-mathieu-schrurs.pdf`. If this
  friction bites, a headless-Chrome export step could automate it later.
- Introduces CV-only content fields that have no home elsewhere in the site.
